API Authentication

Base URL: https://api.acmecorp.com/v2
Last Updated: January 15, 2025

All requests to the Acme Corp API must be authenticated. This guide covers the two supported authentication methods: API keys and OAuth 2.0.

1. API Key Authentication

1.1 Generating an API Key

Account administrators can generate API keys in the Acme Corp dashboard:

  1. Navigate to Settings > API > Keys.
  2. Click Generate New Key.
  3. Assign a descriptive name (e.g., "Production Backend", "CI/CD Pipeline").
  4. Select the permission scope (read-only, read-write, or admin).
  5. Copy the key immediately — it is displayed only once.
Security: API keys are displayed only at the time of creation. Store them in a secrets manager (e.g., HashiCorp Vault, AWS Secrets Manager). Do not commit API keys to source code, store them in environment variables on shared systems, or send them via email or chat.

1.2 Using an API Key

Include the API key in the Authorization header of every request:

GET /v2/data/records HTTP/1.1
Host: api.acmecorp.com
Authorization: Bearer acme_sk_live_7f3a9b2c...

API keys use the prefix acme_sk_live_ for production keys and acme_sk_test_ for sandbox keys. Do not use test keys against the production API or vice versa.

1.3 Key Rotation

API keys should be rotated regularly to reduce the risk of credential compromise. Acme Corp recommends rotating keys every 180 days.

To rotate a key without downtime:

  1. Generate a new key in the dashboard.
  2. Update your application to use the new key.
  3. Verify that requests succeed with the new key.
  4. Revoke the old key in the dashboard.

Both the old and new keys are active simultaneously until the old key is explicitly revoked, giving you time to deploy the change.

1.4 Key Permissions

ScopeAllowed Operations
readGET requests only. Cannot create, update, or delete resources.
read-writeGET, POST, PUT, PATCH, DELETE on resources owned by the account.
adminAll operations, including user management, billing, and API key management.

Follow the principle of least privilege: grant each key only the permissions it needs. CI/CD pipelines typically need read-write; monitoring tools typically need read.

2. OAuth 2.0 Authentication

OAuth 2.0 is the recommended authentication method for applications that act on behalf of a user (e.g., third-party integrations, partner apps). Acme Corp supports two OAuth 2.0 flows:

2.1 Authorization Code Flow (Recommended)

Use the Authorization Code flow for server-side applications that can securely store a client secret.

  1. Redirect the user to the authorization endpoint: 
    GET https://auth.acmecorp.com/oauth/authorize?
  response_type=code&
  client_id=YOUR_CLIENT_ID&
  redirect_uri=https://yourapp.com/callback&
  scope=read+write&
  state=random_csrf_token
  2. User authenticates with their Acme Corp credentials and approves the requested scopes.
  3. Acme Corp redirects back to your redirect_uri with an authorization code: 
    https://yourapp.com/callback?code=AUTH_CODE&state=random_csrf_token
  4. Exchange the code for tokens: 
    POST https://auth.acmecorp.com/oauth/token
Content-Type: application/x-www-form-urlencoded

grant_type=authorization_code&
code=AUTH_CODE&
client_id=YOUR_CLIENT_ID&
client_secret=YOUR_CLIENT_SECRET&
redirect_uri=https://yourapp.com/callback
  5. Receive the token response: 
    {
  "access_token": "acme_at_8f2b...",
  "token_type": "Bearer",
  "expires_in": 3600,
  "refresh_token": "acme_rt_4c7d...",
  "scope": "read write"
}

2.2 Client Credentials Flow

Use the Client Credentials flow for machine-to-machine communication where no user context is needed (e.g., backend service calling the Acme Corp API).

POST https://auth.acmecorp.com/oauth/token
Content-Type: application/x-www-form-urlencoded

grant_type=client_credentials&
client_id=YOUR_CLIENT_ID&
client_secret=YOUR_CLIENT_SECRET&
scope=read

The response contains an access token but no refresh token. Request a new token when the current one expires.

2.3 Token Lifetimes

Token TypeLifetimeRenewable
Access token1 hourNo (request a new one using the refresh token)
Refresh token30 daysYes (each use issues a new refresh token)
Authorization code10 minutesNo (single use)

2.4 Refreshing Tokens

POST https://auth.acmecorp.com/oauth/token
Content-Type: application/x-www-form-urlencoded

grant_type=refresh_token&
refresh_token=acme_rt_4c7d...&
client_id=YOUR_CLIENT_ID&
client_secret=YOUR_CLIENT_SECRET

Each refresh request returns a new access token and a new refresh token. The previous refresh token is invalidated (rotation). If a refresh token is used more than once, all tokens in the chain are revoked as a security precaution.

2.5 Scopes

ScopeDescription
readRead access to the user's resources
writeCreate and update resources
deleteDelete resources
adminAccount administration (user management, billing)
offline_accessRequest a refresh token for long-lived access

Request only the scopes your application needs. Users see the requested scopes during the authorization step and may decline if the request seems excessive.

3. What Happens When Authentication Fails

When a system fails to authenticate against the Acme Corp API, the server returns an HTTP error response with a JSON body describing the failure. The response depends on the type of authentication failure:

3.1 Error Responses

HTTP StatusError CodeDescriptionCommon Cause
401 Unauthorizedinvalid_api_keyThe API key is missing, malformed, or has been revoked.Key was revoked or rotated; application is still using the old key.
401 Unauthorizedexpired_tokenThe OAuth access token has expired.Token lifetime (1 hour) exceeded; application did not refresh.
401 Unauthorizedinvalid_tokenThe OAuth token is malformed or has been revoked.Token was part of a revoked refresh chain; client stored a corrupted token.
403 Forbiddeninsufficient_scopeThe authenticated token does not have the required scope for this endpoint.Token was issued with read scope but the request requires write.
403 Forbiddenip_not_allowedThe request originates from an IP address not in the key's allowlist.IP allowlist configured on the key does not include the requesting server.
429 Too Many Requestsrate_limitedThe API key or token has exceeded the rate limit.Too many requests in the time window. Check the Retry-After header.

3.2 Example Error Response

{
  "error": "invalid_api_key",
  "error_description": "The API key provided is not valid. It may have been revoked or rotated.",
  "status": 401,
  "documentation_url": "https://docs.acmecorp.com/api/auth#errors"
}

3.3 Handling Authentication Failures

Your application should handle authentication failures as follows:

  • 401 with expired_token: Attempt to refresh the token using the refresh token. If the refresh fails, redirect the user to re-authenticate.
  • 401 with invalid_api_key: Check that the key has not been revoked. Generate a new key if necessary. Do not retry with the same key.
  • 403 with insufficient_scope: Request a new token with the required scopes. This may require user re-authorization.
  • 429 rate_limited: Wait for the duration specified in the Retry-After response header before retrying. Implement exponential backoff for repeated 429 responses.
Tip: Log all authentication errors with the x-request-id response header value. When contacting support, include this ID so we can trace the request through our systems.

4. Security Best Practices

  • Store API keys and client secrets in a secrets manager, never in source code or configuration files.
  • Rotate API keys every 180 days.
  • Use IP allowlists to restrict where API keys can be used from.
  • Use the Authorization Code flow (not Client Credentials) when acting on behalf of a user, to ensure proper consent and scope.
  • Request the minimum necessary scopes for each integration.
  • Monitor API key usage in the dashboard under Settings > API > Usage. Set up alerts for unusual patterns (unexpected IP addresses, spike in 401 errors).
  • Revoke keys and tokens immediately if you suspect a compromise.
